udkjs1-作用域

Posted on 2017-07-05   |   In FrontEnd

Overview

读书笔记《你不知道的JavaScript（上卷）》作用域。
基本就是记录下书中提到的重点。

作用域

这本书里对于赋值的操作，做了个有趣的解释，分成了LHS和RHS。

当变量出现在赋值操作的左侧时进行 LHS 查询，出现在右侧时进行 RHS 查询

RHS 查询与简单地查找某个变量的值别无二致，而 LHS 查询则是试图 找到变量的容器本身，从而可以对其赋值。从这个角度说，RHS 并不是真正意义上的“赋 值操作的右侧”，更准确地说是“非左侧”，RHS理解成retrieve his source value(取到它的源值)，这意味着“得到某某的 值”。

作用域嵌套

示例：

function foo(a) { 
    console.log( a + b );
}
var b = 2; 
foo( 2 ); // 4

对 b 进行的 RHS 引用无法在函数 foo 内部完成，但可以在上一级作用域(在这个例子中就 是全局作用域)中完成。

异常

function foo(a) { 
    console.log( a + b ); 
    b = a;
}
foo( 2 );

第一次对 b 进行 RHS 查询时是无法找到该变量的。也就是说，这是一个“未声明”的变 量，因为在任何相关的作用域中都无法找到它。

如果 RHS 查询在所有嵌套的作用域中遍寻不到所需的变量，引擎就会抛出ReferenceError异常。值得注意的是，ReferenceError 是非常重要的异常类型。

相较之下，当引擎执行 LHS 查询时，如果在顶层(全局作用域)中也无法找到目标变量，
全局作用域中就会创建一个具有该名称的变量，并将其返还给引擎，前提是程序运行在非 “严格模式”下。

总结

var a = 2;

以最简单 a = 2为例，

  1. 首先，var a在其作用域中声明新变量。
  2. 接下来，a = 2会查询(LHS查询)变量a并对其进行赋值。
  • 不成功的 RHS 引用会导致抛出 ReferenceError 异常。
  • 不成功的 LHS 引用会导致自动隐式 地创建一个全局变量(非严格模式下)，该变量使用 LHS 引用的目标作为标识符，或者抛 出 ReferenceError 异常(严格模式下)。

词法作用域

JavaScript 中有两个机制可以“欺骗”词法作用域:eval(..) 和 with。前者可以对一段包 含一个或多个声明的“代码”字符串进行演算，并借此来修改已经存在的词法作用域(在 运行时)。后者本质上是通过将一个对象的引用当作作用域来处理，将对象的属性当作作 用域中的标识符来处理，从而创建了一个新的词法作用域(同样是在运行时)

函数作用域与块作用域

函数作用域

var a = 2;
function foo() {
    var a = 3;
    console.log('in foo:'+a);
}

foo();

console.log('out foo:'+a);

结果如图：

foo是个具名函数，在全局作用域中。

var a = 2;
(function foo() {
    var a = 3;
    console.log('in foo:'+a);
})();

console.log('out foo:'+a);

改成IIFE模式，函数可以自动运行，不需要函数名。

IIFE（Immediately Invoked Function Expression）

IIFE 最常见的用法是使用一个匿名函数表达式。

还有一种写法就是把执行的()放在里面 如下：

var a = 2;
(function foo() {
    var a = 3;
    console.log('in foo:'+a);
}());

console.log('out foo:'+a);

  • 有一种进阶的写法，就是传参数进去。

    var a = 2;
(function IIFE( global ) {
        var a = 3;
        console.log( a ); // 3         
        console.log( global.a ); // 2
})( window );

  • IIFE 还有一种变化的用途是倒置代码的运行顺序

    var a = 2;
(function IIFE( def ) { 
    def( window );
})(function def( global ) {
    var a = 3;
    console.log( a ); // 3         
    console.log( global.a ); // 2
});

function def作为参数被传递，并被调用

块作用域

例如：

for (var i=0; i<10; i++) { 
    console.log( i );
}

这种写法，i会被绑定在外部作用域。

try/catch

非常少有人会注意到 JavaScript 的 ES3 规范中规定 try/catch 的 catch 分句会创建一个块作
用域，其中声明的变量仅在 catch 内部有效。

let

ES6新增let关键字， 可以将变量绑定到所在作用域。

const

ES6 还引入了 const，同样可以用来创建块作用域变量，但其值是固定的 (常量)。之后任何试图修改值的操作都会引起错误。

作用域提升

经典：先有鸡还是先有蛋

注意：只有声明本身会被提升，而赋值或其他运行逻辑会留在原地。

例子一： 

a = 2;
var a;
console.log(a); // 2

例子二：

console.log(a); // undefined
var a = 2;

这个例子就印证了只有声明会被提升，赋值逻辑会留在原地。

第二个例子可以拆解为：

var a;
console.log(a);
a = 2;

这个过程就好像变量和函数声明从它们在代码中出现的位置被“移动” 到了最上面。这个过程就叫作提升。

注意：

  • 只有函数声明会被提升，函数表达式不会
  • 每个作用域都会有提升操作
foo(); // 不是 ReferenceError, 而是 TypeError!
var foo = function bar() { 
    // ...
};

变量标识符 foo() 被提升并分配给所在作用域(在这里是全局作用域)，因此 foo() 不会导致 ReferenceError。但是 foo 此时并没有赋值。foo() 由于对 undefined 值进行函数调用而导致非法操作， 因此抛出 TypeError 异常。

函数优先

函数声明和变量声明都会被提升，但函数会优先。

foo(); // 1
    var foo;
function foo() { 
    console.log( 1 );
}
foo = function() { 
    console.log( 2 );
};

这个例子会输出1而不是2,因为引擎是这样理解的。

function foo() { 
    console.log( 1 );
}
foo(); // 1
foo = function() { 
    console.log( 2 );
};

总结

  • var a = 2; 看成两个单独的声明 var a; a = 2;第一个编译阶段，第二个执行阶段
  • 有的声明(变量和函数)都会被“移动”到各自作用域的 最顶端，这个过程被称为提升
  • 声明本身会被提升，而包括函数表达式的赋值在内的赋值操作并不会提升
